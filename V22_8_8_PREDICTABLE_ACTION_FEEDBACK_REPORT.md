# V22.8.8 예측 가능한 행동 피드백 적용 보고서

## 목표

Brunch UX 원칙을 현재 가계부에 필요한 만큼만 적용했습니다. 사용자가 만족한 분석 화면과 이미 안정화된 영수증 흐름은 건드리지 않고, 실제로 오해를 만들던 공통 제출 문구와 비밀번호 확인 흐름을 개선했습니다.

## 발견한 문제

공통 POST 제출 런타임은 로그인, 가입, 참여, 삭제, 탈퇴까지 모두 `저장 중…`으로 바꿨습니다. 중복 제출은 막았지만 사용자가 어떤 동작이 진행 중인지 예측하기 어려웠습니다.

새 계정과 개인 복구 비밀번호 설정은 서버 제출 뒤에야 두 비밀번호의 불일치를 알 수 있었습니다. 모바일에서는 입력을 다시 찾아 수정해야 했고, 상태가 화면 읽기 도구에 미리 전달되지 않았습니다.

## 적용 내용

### 1. 행동별 진행 문구

| 행동 | 진행 문구 |
|---|---|
| 로그인 | `로그인 중…` |
| 새 계정 | `계정 만드는 중…` |
| 가계부 생성 | `가계부 만드는 중…` |
| 초대코드 참여 | `참여 요청 중…` |
| 삭제 | `삭제 중…` |
| 탈퇴·나가기 | `탈퇴 처리 중…` |
| CSV 등 가져오기 | `가져오는 중…` |
| 실제 저장·설정 | `저장 중…` |
| 그 밖의 동작 | `처리 중…` |

button이 아니라 `input[type=submit]`인 경우도 원래 value를 보관하고 복원합니다.

### 2. 제출 상태 접근성·복귀

- 제출된 form과 button에 `aria-busy="true"` 적용
- 보이지 않는 `role="status"`, `aria-live="polite"` 영역에서 진행 문구 알림
- 중복 제출 차단 유지
- 브라우저 `pageshow`에서 버튼 문구, disabled, busy 상태, 실시간 안내 복원
- 복원 후 비밀번호 일치 상태를 다시 계산하도록 form 이벤트 연결

### 3. 위험 작업 확인

- 삭제: 복구하기 어려울 수 있음을 안내
- 탈퇴: 다시 참여하려면 초대가 필요할 수 있음을 안내
- 계정 통합: 계정 연결 정보가 바뀜을 안내
- 기존에 화면별로 더 구체적인 `onsubmit` 확인창이 있으면 이를 우선 유지

### 4. 비밀번호 즉시 피드백

- 새 계정 만들기와 개인 복구 비밀번호 설정에 같은 피드백 런타임 적용
- 초기 빈 상태는 기존 브라우저 `required` 동작 유지
- 입력 시작 후 8자리 미만, 확인 미입력, 불일치를 각각 안내
- 불일치 때 버튼과 native constraint validation으로 제출 차단
- 일치하면 성공 상태를 알리고 제출 허용
- 비밀번호 값 자체는 저장·로그·문서에 남기지 않음
