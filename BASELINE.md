# 유지 기준

후속 버전은 아래 기준을 명시적인 사용자 요청 없이 제거하거나 되돌리지 않습니다.

## 화면과 정보 위계

- 분석 화면의 현재 구성과 렌더링은 고정한다.
- 한 화면의 주 행동을 명확히 하고 보조 행동은 시각적 무게를 낮춘다.
- 실제 사용자 행동과 검증 결과 없이 익숙한 화면을 전면 재배치하지 않는다.
- PC와 모바일을 모두 1차 화면으로 취급한다.
- 모바일 확대를 막는 `maximum-scale=1`, `user-scalable=no`를 사용하지 않는다.
- 자주 쓰는 기능과 관리 기능을 같은 크기·무게로 나열하지 않는다.
- 의미 없는 이중 카드·이중 테두리와 반복되는 동일 리듬을 만들지 않는다.
- 색상 수보다 제목, 간격, 크기, 밀도로 위계를 먼저 만든다.
- 입력 컨트롤은 모바일에서 16px 이상, 주요 터치 대상은 약 44px 이상을 유지한다.

## 로그인과 접근

- PC에서 만든 로컬 계정으로 모바일에서도 같은 로그인 이름과 개인 비밀번호를 사용할 수 있어야 한다.
- 기존 계정 로그인은 기본 동선으로 유지하고 새 계정 만들기는 별도 단계로 둔다.
- 로그인 실패가 새 계정을 자동 생성하지 않도록 유지한다.
- 비밀번호 생성·변경 화면은 8자리와 두 입력의 일치 여부를 제출 전에 안내한다.
- 입력하지 않은 초기 상태는 브라우저의 익숙한 필수 입력 검증을 유지하고, 사용자가 입력을 시작한 뒤에만 불일치를 차단한다.
- 카카오 OAuth의 앱 키·Redirect URI·Client Secret은 운영 콘솔 설정과 일치해야 하며 소스에 Secret을 기록하지 않는다.

## 행동과 상태 피드백

- 버튼의 진행 문구는 로그인, 계정 생성, 참여, 삭제, 탈퇴, 가져오기, 저장의 실제 행동을 반영한다.
- 분류하기 어려운 동작은 `처리 중…`으로 표시하며 저장이 아닌 행동을 `저장 중…`으로 오해시키지 않는다.
- 제출 중 form과 실행 버튼에 `aria-busy`를 표시하고 `aria-live`로 상태를 알린다.
- 뒤로 가기 캐시로 복귀하면 잠금, 진행 문구, 접근성 상태가 원래대로 복원되어야 한다.
- 위험 작업 확인 문구는 `확인` 같은 모호한 표현 대신 삭제·탈퇴·통합의 결과를 설명한다.
- 빈 상태와 오류 상태에는 사용자가 다음에 할 수 있는 행동을 함께 제시한다.
- 애니메이션은 상태 변화나 맥락 연결을 설명할 때만 사용한다.

## 영수증 등록

- 앨범 선택은 카메라 촬영을 강제하지 않는다.
- 카메라 촬영은 별도 버튼으로 제공한다.
- 사진 선택만으로 OCR을 자동 실행하지 않는다.
- 사진은 OCR 목적으로 서버에 업로드하지 않고 브라우저 안에서 처리한다.
- 미리보기 후 사용자가 명시적으로 사진 읽기를 실행한다.
- 자동 인식값은 원본과 비교한 뒤 확인 체크를 해야 저장할 수 있다.
- 같은 영수증의 중복 저장과 조회 전용 권한의 저장을 차단한다.
- OCR 실패·오프라인·시간 초과·사용자 취소 후 화면과 버튼이 복구되어야 한다.

## 성능과 데이터 안정성

- OCR 라이브러리는 영수증 화면 진입 시 미리 다운로드하지 않는다.
- 큰 사진은 OCR 전에 축소하고 연속 작업에서는 초기화된 작업자를 재사용한다.
- 영수증 화면에서 멤버·달력·예산 등 사용하지 않는 전체 데이터를 조회하지 않는다.
- 한 요청 안의 동일한 세션 검증을 반복 조회하지 않는다.
- 모바일 월간 데이터는 한 번 조회한 결과를 화면별로 재사용한다.
- 분석 화면의 정적 스크립트 요청에서 불필요한 계정 병합 조회를 반복하지 않는다.

## 카카오 1:1·그룹 채팅

- `botGroupKey`가 있는 요청은 그룹 채팅으로 취급한다.
- 그룹 응답에는 QuickReplies, CommerceCard, Carousel을 보내지 않는다.
- 그룹에서 빠른 선택이 필요하면 사용자가 그대로 입력할 수 있는 짧은 번호형 문장으로 제공한다.
- 1:1 채팅의 단계형 QuickReplies는 별도 동작으로 유지한다.
- 그룹 응답 output은 SimpleText, SimpleImage, TextCard, BasicCard, ListCard, ItemCard만 사용하고 한 응답을 짧게 유지한다.
- SkillRequest에 알 수 없는 필드가 추가되어도 실패하지 않는다.
- 사용자 식별은 `botUserKey`, `appUserId`를 우선한다. `plusfriendUserKey`는 기존 데이터 호환용 최후순위 폴백 외 신규 로직에서 사용하지 않는다.
- 그룹방 전체 대화를 읽거나 학습한다고 표현하지 않는다. 챗봇에게 직접 전달된 명령만 처리한다.
- URL 쿼리로 전달된 식별자만으로 로그인이나 가계부 권한을 확정하지 않는다.
- Event API, 선톡, 자동 푸시는 명시적인 기능 승인과 최신 카카오 정책 재확인 없이 도입하지 않는다.

## 병합 원칙

- 외부 수정본은 전체 덮어쓰기가 아니라 현재 기준본과 비교해 선별 반영한다.
- 기능이 중복되면 더 안전하고 단순한 한 구현만 남긴다.
- 배포 전에 로그인·홈·메뉴·분석·영수증과 모바일 확대 허용 여부를 함께 확인한다.
- 매 버전에서 신규 SQL, Worker 전체 교체, 환경변수, OpenBuilder 변경 필요 여부를 각각 판정해 `DEPLOYMENT_MATRIX.md`에 기록한다.
- UX 변경은 `docs/ux/UX_PRINCIPLES_BRUNCH_110_PROJECT_MEMORY.md`의 판단 순서를 따르되, 사용자의 명시적인 화면 선호를 최우선으로 보호한다.
