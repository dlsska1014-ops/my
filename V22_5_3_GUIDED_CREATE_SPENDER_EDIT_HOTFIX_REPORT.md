# V22.5.3 생성 단계·지출자 수정 안정화 보고서

## 기준 버전

- 이전: `V22.5.2-SLASH-REPRESENTATIVE-COMMAND-ALIGNMENT`
- 현재: `V22.5.4-CONVERSATION-STATE-INTEGRITY-HARDENING`

## 확인된 원인

### 1. 생성 확인 응답의 우선순위 누락

생성 확인 단계가 `네`, `예`, `확인` 등 일부 표현만 허용했습니다. `응`, `만들어줘`는 확인으로 판정되지 않았고, 뒤의 일반 이름 판정기가 이를 새 가계부 이름으로 해석했습니다.

### 2. 숫자형 종류 선택 미지원

종류 안내에는 1~5번이 표시되지만 파서는 `가족`, `여행`, `직접 입력` 같은 텍스트만 인식했습니다. 따라서 `5번`이 종류가 아니라 이름 후보로 넘어갔습니다.

### 3. 거래 수정 항목에 지출자 미포함

챗봇 번호 수정은 금액·분류·결제수단·내용·날짜·삭제만 지원했습니다. `지출자 변경`은 거래 수정 의도로 라우팅되지 않아 일반 입력 예시로 떨어졌습니다.

## 반영 사항

- `1`, `1번`, `첫 번째`부터 `5`, `5번`, `다섯 번째`까지 종류 선택 지원
- 생성 긍정 응답: `응`, `ㅇㅇ`, `네`, `예`, `좋아`, `그래`, `맞아`, `만들어줘`, `생성해줘`, `진행해줘` 등 지원
- 생성 부정 응답: `아니`, `아니요`, `다른 이름`, `이름 바꾸기` 등 지원
- 선택번호·확인어·취소어를 가계부 이름으로 저장하지 않음
- 종류를 선택한 뒤 이름을 입력해도 즉시 생성하지 않고 최종 확인 수행
- `지출자 변경`은 방금 저장한 본인 거래를 선택하고 구성원 목록을 제공
- `01번 지출자 엄마`, `방금 지출자 엄마로 변경` 직접 수정 지원
- 동일·유사 이름이 여러 명이면 번호로 재선택
- owner/admin/member만 지출자 후보로 제공
- 지출자 변경 후 응답에 가계부·거래·변경된 지출자 표시
- 기존 구성원 조회를 병렬 처리하여 수정 단계 응답 지연 완화

## 유지 사항

- 단톡방 1개당 연결 가계부 1개 규칙
- 직접 채팅 선택 가계부와 단톡방 연결 가계부 분리
- 거래 저장 응답의 가계부 이름 표시
- V22.5.2 슬래시 호환
- V22.5.1 가계부 컨텍스트 안전 규칙
- 공개 홈페이지·Search Console·AdSense·NLU 운영 기능
- OpenBuilder 시나리오명·블록명·Skill URL
- Supabase 기존 스키마

## 데이터 처리 기준

지출자 변경은 기존 데이터 모델의 `transactions.user_id`를 선택한 구성원 ID로 변경합니다. 따라서 요약·정산에서는 변경된 구성원의 지출로 집계됩니다.
