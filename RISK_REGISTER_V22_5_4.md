# V22.5.4 잔여 위험과 운영 대응

V22.5.4는 확인된 상태 전이·권한·동시성 오류를 코드와 자동시험으로 줄였지만, 실제 운영에서 발생 가능한 모든 조합을 수학적으로 보장하지는 않습니다.

## 배포 후에도 남는 위험

| 위험 | 영향 | 현재 대응 | 추가 대응 시점 |
|---|---|---|---|
| Cloudflare 여러 인스턴스에 동일 생성 확인이 거의 동시에 도착 | 동일 이름 가계부가 2개 생길 가능성 | 흐름 확인·사용자별 중복 조회 | 실제 발생 시 DB RPC/idempotency key 도입 |
| DB 저장은 완료됐지만 카카오 응답 전에 네트워크가 끊김 | 사용자가 실패로 오해해 재전송 | DB 최근 동일 거래 중복검사 | 운영 로그에서 반복되면 요청 ID 영구 저장 |
| Kakao 그룹 payload에서 방 관리자 여부 미제공 | 실제 방 관리자 판별 불가 | 가계부 owner/admin 양쪽 권한으로 재연결 제한 | Kakao 공식 payload가 방 관리자 정보를 제공할 때 연동 |
| 웹 로그인 계정과 botUserKey 계정이 분리 | 서로 다른 가계부 목록 노출 | 자동 병합 금지, `/identity-audit` 명시적 통합 | 사용자 확인 후만 통합 |
| Supabase RLS·컬럼·RPC가 배포 환경과 다름 | 일부 경로 오류 | 안전 응답·진단 화면·회귀시험 | 배포 전 SQL 감사 및 운영 smoke |
| 과거 `kakao_group_links` 단일 JSON | 동시 저장 유실 가능성 | V22.5.4 신규·재연결은 방별 설정에 저장하고, 기존 값은 병렬 호환 조회·미러 유지 | 충분한 운영 기간 후 legacy mirror 제거 검토 |
| 5,000개 초과 단톡방 연결 | 관리자 전체 목록/일괄 정리 일부 누락 가능 | 실제 메시지 처리는 방별 직접 조회 | 규모 도달 전 전용 테이블/RPC로 이전 |
| 실제 iOS·Android·OpenBuilder UI 차이 | 메뉴 표시·줄바꿈 차이 | 대표 명령어 계약 JSON과 최소 실기기 확인 | 관리자센터 변경 시 재확인 |
| 대량 동시 사용자 | 응답 지연·rate limit | 사용자별 rate limit, 병렬 조회, 450ms 지연시험 | 실제 트래픽 전 부하시험·Queues 검토 |

## 운영 원칙

- 계정은 자동 병합하지 않습니다.
- 가계부와 거래는 자동 삭제하지 않습니다.
- 단톡방은 방 하나당 가계부 하나만 연결합니다.
- 오류가 의심되면 동일 문장을 반복 전송하기보다 `오늘 기록 보기`와 `/nlu-ops`를 먼저 확인합니다.
- 새 오류는 재현 발화를 평가 데이터와 smoke에 추가한 뒤 배포합니다.
