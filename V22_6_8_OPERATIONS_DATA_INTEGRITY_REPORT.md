# V22.6.8 운영·데이터 무결성 안정화 보고서

## 결과

현재 릴리스 식별자는 `V22.6.8-OPERATIONS-DATA-INTEGRITY`입니다. V22.6.7의 UI·UX, 무료 공개 기능, 자연어 가져오기, 계정 통합, 다중 가계부·단톡방 격리, 미완성 밈/카드 실적 숨김 정책을 유지하면서 동시 저장과 예약 작업의 데이터 무결성을 보강했습니다.

## 구현 내용

### 영수증

- 같은 인스턴스의 동일 영수증 요청을 `가계부 + 날짜 + 금액 + 원문 지문` 단위로 직렬화합니다.
- DB에는 가계부·날짜·금액·정규화 원문(원문이 없으면 상호) 고유 인덱스를 둡니다.
- 다른 인스턴스에서 동시에 저장되어 고유 충돌이 발생하면 사용자에게 오류 대신 “중복 저장 방지”를 안내합니다.
- 원문이 서로 다른 정상 영수증을 상호·금액만으로 합치지 않도록 원문이 있을 때는 원문을 우선 비교합니다.

### 정기거래

- 자동 거래의 식별자를 `recurring:{규칙ID}:{YYYY-MM}`로 고정합니다.
- DB 고유 인덱스 충돌을 실패가 아닌 중복 확인으로 처리하고, 거래가 이미 있는 경우에도 해당 규칙의 `last_applied_month`를 갱신합니다.
- 거래 저장 또는 기존 거래 확인이 끝나기 전에는 적용월을 변경하지 않습니다.
- 한 가계부·규칙이 실패해도 다른 가계부와 규칙은 계속 처리하며 `failed`와 운영 이벤트에 남깁니다.

### 정산 완료 이력

- 정산 이력은 설정 JSON 한 행을 읽고 다시 쓰므로 `settlement-history:{household_id}` lease로 가계부 단위 직렬화합니다.
- 동시 요청이 잠금을 얻지 못하면 기존 데이터를 덮어쓰지 않고 재전송 방지 안내를 표시합니다.
- 잠금을 얻은 뒤 최신 이력을 다시 읽고, 같은 사용자·월·방식·금액·메모의 5분 이내 완료를 중복 제외합니다.

### Cron과 재시도

- 정기거래는 `cron:recurring:{YYYY-MM}`, 리포트는 `cron:reports:{YYYY-MM-DD}` DB lease를 사용합니다.
- 동일 작업이 실행 중이면 쓰기 없이 `ok: true`, `skipped_run: true`, `reason: already_running`을 반환합니다.
- 기본 lease는 15분이며 Worker가 중단돼도 만료 후 재시도할 수 있습니다.
- DB RPC가 아직 없을 때는 같은 Worker 인스턴스 안에서만 임시 잠금을 사용하고 `lock_mode: memory-fallback`과 경고 이벤트를 남깁니다.

### 호환 마이그레이션과 감사

- `OPERATIONS_DATA_INTEGRITY_AUDIT_V22_6_8.sql`은 중복, 소유자 이상, 고아 참여·거래, 설정 범위 후보를 조회만 합니다.
- `schema_v22_6_8_operations_integrity.sql`은 관련 중복이 남아 있으면 예외로 중단하며 기존 행을 임의 수정·삭제하지 않습니다.
- 설정 키, 카카오 식별키, 참여행, 초대코드, 영수증, 정기거래에 고유 인덱스를 추가합니다.
- `accountbook_operation_locks`와 서비스 역할 전용 claim/release 함수를 추가합니다.

## UI·UX 유지 확인

- 홈 상단 분석·메뉴와 하단 5개 메뉴 유지
- 보안 설정 완료 후 안전한 이전 경로 복귀 유지
- 가계부 이름·카카오 표시명·개인 비밀번호 → 생성 → 초대 순서 유지
- 가계부별 이름 변경·참여자·단톡방·예산·백업·영구 삭제 옵션 유지
- 실제 수입 자동 합계, 예상 수입 종류별 합계, 지출 분류별 예산 합계 유지
- 정산 저장 경합 시 “반복 제출하지 말고 잠시 후 이력을 확인”하는 이탈 방지 문구 추가
- 360×780과 390×844에서 18개 경로씩 검사해 가로 넘침, 화면 밖 요소, 중복 ID, 입력 라벨 누락, 숨김 기능 링크, 푸터/하단 메뉴 겹침이 모두 0건

## 공개 정책

- 영수증·정기거래·정산·리포트·분석·예산·백업/가져오기는 무료 일반 기능입니다.
- 밈/소비 카드와 카드 실적/혜택은 계속 숨기며 전용 QA 잠금 없이 직접 URL도 열리지 않습니다.
- 결제·구독·사용량 제한은 연결하지 않았습니다.

## 배포 전제

이 번들은 운영 DB에 자동 적용하거나 Worker를 자동 배포하지 않습니다. 백업 → 조회 전용 감사 → 중복 검토 → 마이그레이션 → Worker 배포 → 수동 검증 순서로 적용해야 합니다.
