# UX 원칙 프로젝트 메모

- 학습 원문: https://brunch.co.kr/@dalgudot/110
- 프로젝트 반영일: 2026-07-20
- 적용 시작 버전: `V22.8.8-PREDICTABLE-ACTION-FEEDBACK`
- 성격: 원문을 복제한 문서가 아니라 가계부 프로젝트에서 반복 사용할 판단 기준을 요약한 메모

## 학습한 핵심 원칙

1. 제작자는 사용자가 아니다. 추측보다 실제 행동, 문의, 오류, 검증 결과를 먼저 본다.
2. 한 화면의 주 행동은 하나로 명확히 하고 보조 행동은 시각적 무게를 낮춘다.
3. 로딩·성공·실패·취소·복귀 상태를 즉시 알려 사용자가 다음 상태를 예측할 수 있게 한다.
4. 사용 흐름과 맥락을 끊지 않는다. 완료 뒤 어디로 가는지, 오류 뒤 무엇을 할 수 있는지 연결한다.
5. 같은 의미의 요소는 같은 문구·색·간격·상태 규칙을 사용한다.
6. 빈 결과와 오류도 정상 상태로 설계하고 다음 행동을 함께 제시한다.
7. 움직임은 상태 변화나 공간 관계를 설명할 때만 사용한다.
8. 버튼과 안내 문구는 `확인` 같은 모호한 표현 대신 실제 행동을 말한다.
9. 익숙한 위치와 패턴을 존중하고 이동이 꼭 필요하면 전환 안내를 제공한다.
10. 핵심 가치에 필요하지 않은 요소는 줄이되, 사용자가 판단하는 데 필요한 정보까지 숨기지 않는다.

핵심 철학은 UX를 장식이 아니라 사용자가 불안하거나 헷갈리지 않도록 미리 배려하는 일로 보는 것입니다.

## 똑똑한 가계부 적용 규칙

### 변경 전에

- 사용자의 명시적인 만족 화면과 보호 요청을 먼저 확인한다.
- 실제 재현 경로, 기기, 네트워크, 권한, 서버 응답을 확인한다.
- 새 기능보다 기존 흐름에서 오해·중복·막힘이 발생하는 지점을 찾는다.
- 변경이 SQL, `src/index.js`, 환경변수, OpenBuilder 중 어디에 필요한지 각각 판정한다.

### 화면을 바꿀 때

- 주 행동은 한 문장으로 설명할 수 있어야 한다.
- 버튼은 `로그인`, `참여`, `삭제`, `저장`처럼 결과가 보이는 동사를 사용한다.
- 진행 문구도 `로그인 중…`, `삭제 중…`처럼 같은 동사를 유지한다.
- 중요한 결과는 확인창, 가벼운 완료는 토스트나 인라인 상태를 사용한다.
- 오류와 빈 상태에는 원인만 말하지 않고 다시 시도하거나 이동할 경로를 제시한다.
- 익숙한 분석 화면이나 메뉴를 근거 없이 재배치하지 않는다.

### 검증할 때

- 마우스뿐 아니라 키보드, VoiceOver, TalkBack에서 상태가 전달되는지 본다.
- 제출 중 중복 실행 차단, 취소, 서버 오류, 뒤로 가기 복귀를 함께 확인한다.
- 모바일 확대와 터치 대상 크기를 보호한다.
- 보호 화면은 경로 응답뿐 아니라 소스 해시 또는 고정 픽스처로 변경 여부를 검증한다.
- 자동 검증과 실제 기기 확인을 분리해 완료 여부를 기록한다.

## V22.8.8에서 실제 적용한 항목

| 원칙 | 적용 |
|---|---|
| 행동별 명확한 문구 | 모든 POST 버튼의 일괄 `저장 중…`을 행동별 진행 문구로 변경 |
| 상태 피드백 | form·button `aria-busy`, 보조기기용 `aria-live` 상태 추가 |
| 오류 예방 | 두 비밀번호의 8자리·불일치·일치를 입력 중 안내 |
| 결과 설명 | 삭제·탈퇴·계정 통합 확인창에 실제 영향 표시 |
| 익숙함 보호 | 사용자 만족 대상인 분석 화면을 V22.8.7 해시로 고정 |
| 필요한 만큼만 변경 | 이미 안정적인 영수증 화면은 재설계하지 않고 기존 회귀 검증 유지 |

## 계속 보호할 결정

- 분석 화면은 별도 요청 없이 변경하지 않는다.
- 기존 계정 로그인을 주 동선으로, 새 계정 만들기를 보조 동선으로 유지한다.
- 영수증은 사진 선택 → 명시적 OCR → 원본 확인 → 저장의 순서를 유지한다.
- 카카오 그룹과 1:1 응답 규격을 섞지 않는다.
- 데이터로 확인되지 않은 장기 행동 추적과 A/B 실험은 개인정보·SQL 계획 없이 추가하지 않는다.
