## V22.8.10 실제 적용 판단

### Cloudflare Worker

- 적용 파일: `src/index.js`
- 방식: 기존 Worker 코드 전체 교체
- 부분 붙여넣기: 금지
- 새 정적 경로 두 개는 교체된 Worker가 직접 제공합니다.
  - `/assets/mobile-home-v22810.css`
  - `/assets/mobile-home-v22810.js`

### Supabase SQL

- 신규 SQL 파일: 없음
- 실행 작업: 없음
- 기존 SQL 재실행: 불필요
- 새 테이블·컬럼·RLS·RPC 참조: 없음
- 기존 필드만 사용: 각 테이블의 `id`, `accountbook_settings.key`, `household_id`, `month`, `transaction_date`
- 가계부·사용자 `id` 묶음 조회와 settings `key` 묶음 조회는 기존 기본키·V22.6.8 key unique 기준을 그대로 사용합니다.
- `household_members.user_id` 및 월간 거래 인덱스 추가는 이번 버전의 필수 조건이 아닙니다. 운영 실행 계획에서 실제 순차 스캔 병목이 확인될 때만 별도 멱등 SQL로 검토합니다.

### Cloudflare 환경변수·Secret

- 신규 환경변수·Secret: 없음
- 기존 Supabase, 세션, 카카오 로그인 설정을 그대로 사용
- 실제 Secret 값은 ZIP에 포함하지 않음

## 다음 버전 판단 규칙

1. 새 테이블·컬럼·인덱스·RLS·RPC 또는 기존 쿼리의 새 필드 참조를 먼저 검색합니다.
2. DB 변경이 있으면 누적·멱등 SQL, 적용 순서, 검증 쿼리, 롤백 한계를 먼저 작성합니다.
3. DB 참조가 없으면 SQL을 만들지 않고 `src/index.js` 교체만 명시합니다.
4. 속도 문제 때문에 인덱스를 검토할 때는 운영 행 수, 쿼리 로그, `EXPLAIN (ANALYZE, BUFFERS)` 근거를 먼저 확보합니다.
5. 환경변수, Kakao Developers, OpenBuilder 작업을 각각 별도 판정합니다.
6. 보호 화면은 소스 해시와 경로 회귀 검증으로 변경 여부를 확인합니다.
7. 최종 ZIP의 매트릭스, 실제 파일, 검증 보고서, 체크섬을 서로 대조합니다.
