# 계정·가계부 안정화 런북

## 1. 배포

1. V22.4.1 Worker 배포
2. `/health`에서 아래 버전 확인

```text
V22.4.1-IDENTITY-HOUSEHOLD-STABILITY-HOTFIX
```

## 2. 데이터 변경 전 감사

Supabase SQL Editor에서 `identity_audit_v22_4_1.sql`을 실행합니다.

확인 항목:

- 같은 `kakao_user_key`가 중복된 사용자
- 같은 가계부·사용자의 중복 참여 행
- 소유자가 2명 이상인 가계부

조회 SQL은 데이터를 수정하지 않습니다.

## 3. 관리자 화면 점검

관리자 로그인 후:

```text
https://ttokttok-accountbook.com/identity-audit
```

문제가 있는 가계부를 선택합니다.

각 계정에서 확인할 정보:

- 표시 이름
- 계정 유형
- 역할
- 해당 가계부 거래 건수

## 4. 계정 통합

주 계정은 앞으로 사용할 로그인 계정으로 선택합니다.

권장 판단 순서:

1. 현재 웹에서 정상 로그인되는 계정
2. 거래 건수가 많은 계정
3. 유지해야 할 표시 이름과 권한이 있는 계정

보조 계정은 주 계정으로 합칠 계정입니다.

확인 문구 `통합`을 입력한 뒤 실행합니다.

주의:

- 화면만 보고 어느 계정이 주 계정인지 확신할 수 없으면 실행하지 않습니다.
- 통합 후 보조 계정으로 로그인 중인 브라우저는 로그아웃 후 주 계정으로 다시 로그인합니다.
- 통합은 거래와 가계부를 삭제하지 않지만, 운영 데이터 백업 후 실행하는 것을 권장합니다.

## 5. 통합 후 확인

- `/households`에서 소유자 1명인지 확인
- 거래 건수가 유지되는지 확인
- 카카오톡 자연어 거래 1건 저장
- 웹 로그인 후 같은 가계부가 보이는지 확인
- 초대코드가 기존과 같은지 확인

## 6. 제약조건 적용

중복 정리가 끝난 뒤에만 `schema_v22_4_1_identity_constraints_optional.sql`을 실행합니다.

이 SQL은 중복 데이터가 남아 있으면 오류를 내고 중단합니다.

적용되는 제약:

- `users.kakao_user_key` 중복 방지
- `household_members(household_id,user_id)` 중복 방지

## 7. 금지 사항

- 문제 계정을 먼저 삭제하지 않기
- 거래를 다른 가계부로 수동 이동하지 않기
- 초대코드를 재발급해서 문제를 우회하지 않기
- 소유자 두 명 중 한 명을 근거 없이 삭제하지 않기
