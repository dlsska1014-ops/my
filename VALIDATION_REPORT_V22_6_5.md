## 모바일 렌더링 검증

다음 13개 경로를 한글 웹폰트로 390×844와 360×780에서 각각 렌더링했다.

`/app`, `/my/households`, `/my/members`, `/budgets`, `/my/settings`, `/my/backup-login`, `/menu`, `/my/backup`, `/my/analysis`, `/my/premium`, `/payment-methods`, `/reserve-plans`, `/settlement-summary`

두 해상도 모두 다음 조건을 만족했다.

- 문서 너비가 viewport 너비와 동일
- viewport 밖으로 튀어나온 표시 요소 0개
- 중복 DOM ID 0개
- 이름 없는 입력·select·textarea 0개
- 사업자 정보 footer 존재
- 고정 하단 메뉴가 있는 화면에서 footer와 96~132px 간격 확보

첫 렌더링에서 한글 폰트 기준 빠른 입력 행이 오른쪽으로 넘치는 문제를 발견했고, grid의 첫 열과 input에 `min-width:0`을 적용한 뒤 재검증했다.

## 실행 환경 한계

- 렌더링은 모바일 Chromium 모의 환경이며 iPhone Safari 자체 엔진 검증은 아니다.
- 운영 Supabase의 실제 RLS·외래키·트리거·용량·장애 시나리오는 배포 환경이 필요하다.
- 실제 카카오 OAuth 닉네임 응답과 OpenBuilder 단톡방 연결은 카카오 운영 설정이 필요하다.
- 실제 타 가계부 파일의 모든 변형을 보장할 수 없으므로 미인식 사유 화면과 소량 미리보기를 계속 사용해야 한다.
- 가계부 삭제는 여러 테이블을 순서대로 정리하므로 운영 DB 백업과 테스트 가계부 삭제 확인이 필수다.
