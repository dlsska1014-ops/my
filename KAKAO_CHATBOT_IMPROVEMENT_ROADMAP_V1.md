## 3. P0 — 운영 배포 전 반드시 해결할 항목

### P0-1. 비정상 요청의 테스트 사용자 저장 경로 차단

현재 `/skill`은 JSON 파싱에 실패해도 본문 문자열을 테스트 발화로 변환하고 고정된 테스트 사용자 키로 내부 처리한다. 공개 운영 주소에 임의 POST가 들어오면 테스트 사용자 데이터 오염이나 불필요한 쓰기가 발생할 수 있다.

개선 기준:

- 운영 기본값에서는 비 JSON 본문을 거래 처리하지 않는다.
- 원문 테스트 허용은 `ALLOW_RAW_SKILL_TEST=1` 같은 명시적 개발 환경 설정에서만 가능하게 한다.
- 운영에서는 HTTP 200의 안전 응답만 반환하고 저장·수정·참여 로직에 진입하지 않는다.
- 가능하면 OpenBuilder가 지원하는 요청 인증 헤더를 공식 문서로 확인한 후 별도 적용한다.

합격 조건:

- 일반 텍스트 POST로 Supabase 쓰기 요청이 0회 발생한다.
- 정상 카카오 JSON Payload는 기존과 동일하게 동작한다.

### P0-2. quickReplies 기본 자동 삽입 제거

코드의 `kakaoText()`는 응답마다 quickReplies를 자동 생성하고 `/health`도 `enabled`로 표시한다. 반면 확정 기준서와 XLSX 설정 기준은 기본 미사용이다.

개선 기준:

- `kakaoText()` 기본 응답은 `template.outputs`만 포함한다.
- quickReplies 자동 생성 함수는 호출하지 않는다.
- `/health`와 명령어 JSON의 상태를 `disabled_by_baseline`으로 통일한다.
- 향후 공식 지원 여부를 확인해도 별도 버전과 명시적 설정 없이는 다시 켜지 않는다.

합격 조건:

- 모든 기본 `/skill` 응답에 `template.quickReplies`가 없다.
- 응답은 말풍선 1개, `simpleText` 1개다.

### P0-3. 정식 용어를 `남은예산`으로 통일

XLSX에는 `오늘예산`, `오늘 예산`이 있고 테스트 문장도 이를 사용하지만 실제 예산 파서는 해당 표현을 인식하지 않는다. 코드 메뉴와 quickReplies에도 `오늘예산`이 남아 있다.

개선 기준:

- 블록명은 `31_남은_예산`으로 변경한다.
- 대표 명령어·블록 발화·도움말·테스트 문장을 `남은예산` 중심으로 통일한다.
- `오늘예산`, `오늘 예산`은 사용자 노출 목록에서 제거한다.
- 호환 별칭을 유지할지 여부는 별도 결정하되, 정식 용어로 노출하지 않는다.

합격 조건:

- 코드·문서·XLSX 사용자 노출 영역에서 `오늘예산`이 0건이다.
- `남은예산`이 예산 응답으로 라우팅된다.

### P0-4. 가계부 만들기·참여·전환을 안내형으로 제한

현재 코드와 XLSX는 카카오톡 안에서 새 가계부 생성, 초대코드 참여, 목록 전환을 직접 처리한다. 확정 기준서는 파서·권한 검증 전까지 안내 블록으로 시작하도록 정했다.

개선 기준:

- `가계부 만들기`, `가계부 참여`, `가계부 전환`은 데이터 변경 없이 안내만 반환한다.
- 응답에는 대표 주소 `https://ttokttok-accountbook.com/my/households` 한 개만 포함한다.
- `단톡방 연결 (+초대코드)`은 기존 지원 기능으로 유지한다.
- 직접 처리 코드는 V21.5 설계 검증 전까지 사용자 경로에서 비활성화한다.

합격 조건:

- 위 세 명령으로 household/member 쓰기 요청이 발생하지 않는다.
- 단톡방 연결 회귀 테스트는 통과한다.

### P0-5. URL 노출 정책 정렬

현재 도움말·예산·입력예시·키워드·수정·저장 결과·요약 응답 등에 URL이 자동으로 붙는다. 기준서는 사용자가 링크를 요청했을 때만 대표 주소 1개를 보내도록 정했다.

개선 기준:

- 일반 저장·조회·도움말 응답에서 URL을 제거한다.
- `링크`, `주소`, 명시적 웹 이동 요청에만 대표 주소 1개를 반환한다.
- 가계부 만들기·참여·전환 안내는 해당 기능 요청 자체가 웹 이동 요청이므로 `/my/households` 한 개를 허용한다.

합격 조건:

- 일반 저장·요약·남은예산·오늘 기록 응답 URL 수는 0개다.
- 링크 요청 응답 URL 수는 정확히 1개다.

### P0-6. OpenBuilder 블록 수를 확정 구조에 맞춤

현재 XLSX의 23블록 구조는 확정 기준서의 기존 12블록 유지 원칙보다 과도하게 분리되어 있다.

개선 기준:

- 기존 12개 업무 블록을 유지한다.
- 자유 거래문은 별도 예시를 대량 등록하지 않고 폴백 → `/skill`로 전달한다.
- 가계부 만들기·참여·전환은 하나의 안내 블록으로 통합 검토한다.
- 시스템 블록은 업무 블록 수와 별도 집계한다.
- 파라미터·컨텍스트는 기본 없음으로 유지한다.

합격 조건:

- XLSX 요약의 블록 수 정의가 명확하다: `시스템 블록`, `업무 블록`, `합계`를 분리한다.
- 동일 발화 중복은 0건이다.
- 폴백 블록 `/skill` 연결이 필수 체크 항목으로 남는다.

### P0-7. 정산·분류 조회의 파서 정합성 보완

XLSX에는 `모임비`, `모임 정산`, `여행 정산`, `카테고리별 지출`, `항목별 지출`이 있으나 현재 파서 정규식과 완전히 일치하지 않는다. 또한 `정산`은 월 요약과 동일 처리되어 사용자 기대와 다를 수 있다.

개선 기준:

- V21.4.4에서는 지원 의미가 불명확한 별칭을 OpenBuilder에서 제거하거나 안내 응답으로 제한한다.
- `정산`과 `정산현황`의 실제 의미를 먼저 정의한다.
- 금액 변경 또는 정산 확정 기능은 명시적 패턴과 확인 단계 없이는 실행하지 않는다.

합격 조건:

- XLSX에 등록된 모든 발화마다 대응 파서·예상 응답·부작용 여부가 명시된다.
- 파서가 없는 발화를 처리형 블록으로 노출하지 않는다.

### P0-8. 버전·패키지 이름 정리

개선 기준:

- ZIP명, 내부 루트 폴더명, `VERSION.txt`, `APP_VERSION`, 버전 보고서, XLSX 버전을 동일하게 맞춘다.
- 권장 다음 버전: `V21.4.4-BASELINE-ALIGNMENT-HOTFIX`.
- 패키지 안에 `KAKAO_CHATBOT_BASELINE_V1.md`와 이번 개선 보고서를 포함한다.

합격 조건:

- 버전 문자열 정적 검사에서 과거 루트 버전 혼입이 없다.

### P0-9. XLSX 가독성과 수량 정합성 개선

현재 렌더링에서는 `Carlito` 폰트의 한글 표시가 불안정하고, 여러 발화를 한 셀에 넣어 행 높이가 과도하게 커진다.

개선 기준:

- 한글 지원 글꼴로 통일한다.
- 한 행에는 한 발화만 두고, 블록 요약은 수식 또는 생성 스크립트로 집계한다.
- `적용 상태`, `테스트 결과`, `우선순위`는 드롭다운으로 관리한다.
- 배포 스냅샷 ID, 작업자, 확인 일시, 실제 결과 열을 추가한다.

합격 조건:

- 모든 시트를 실제 렌더링해 한글 누락·잘림·과도한 행 높이가 없다.
- 요약 수량과 상세 행 수가 일치한다.

## 5. V21.5 — 가계부 관리 기능의 제한적 재설계

직접 처리 기능은 한 번에 모두 열지 않는다.

### 권장 순서

1. 가계부 목록 조회
2. 선택 가계부 변경
3. 초대코드 참여
4. 새 가계부 생성

각 단계는 이전 단계의 실제 그룹방 검증 완료 후 진행한다.

### 필수 조건

- owner > admin > member > viewer > pending 권한 우선순위를 유지한다.
- 카카오 고유 사용자 키는 변경하지 않고 가계부 내 표시명만 변경한다.
- 변경 명령은 명확한 확인 응답을 제공한다.
- 실패 시 중간 상태가 남지 않도록 한다.
- 선택 가계부와 진행 상태는 Supabase에 저장한다.
- OpenBuilder 컨텍스트는 영구 상태로 사용하지 않는다.

- 버전 문자열 일치
- 금지 문법·개인 주소·과거 버전명 검사
- ZIP 무결성 및 ZIP 내부 재검사

### 스킬 계약 검사

- HTTP 200
- `version: "2.0"`
- `template.outputs` 1개 이상
- 일반 응답 `simpleText` 1개
- 기본 quickReplies 없음
- simpleText 안전 길이 이하
- 명시적 링크 요청 외 URL 없음

### 기능 회귀

- 기록·입력 접두어 스트리핑
- 수정 접두어 스트리핑
- 오늘 기록 조회
- 남은예산
- 요약
- 단톡방 연결
- 반복 발화 방어
- 거래 단위 중복 방어
- owner/admin/member/viewer/pending 권한
- 여러 가계부 범위 분리
- 캘린더 리다이렉트

### 부정 테스트

- 비 JSON POST는 저장하지 않음
- 오늘예산은 사용자 문서에 노출되지 않음
- 새 가계부 만들기·참여·전환은 P0 버전에서 DB를 변경하지 않음
- viewer/pending이 저장·수정·삭제하지 못함
- 같은 발화가 여러 블록에 중복되지 않음
- 파서가 없는 발화가 처리형 블록에 등록되지 않음

### 실사용 검사

개발 채널 → 실제 그룹방 → 운영 배포 순서로 진행한다.

실제 그룹방에서는 다음을 기록한다.

- 테스트 일시
- 테스트 사용자 역할
- 발화
- 예상 블록
- 실제 응답
- DB 변경 여부
- URL 개수
- 합격/불합격
- 증빙 스크린샷 또는 로그 ID

## 9. 배포 게이트

### Worker 배포 전

- 현재 운영 패키지와 환경변수 목록을 별도 스냅샷으로 기록한다.
- DB 스키마 변경 없음 여부를 확인한다.
- 전체 자동 테스트를 통과한다.
- 개발 URL에서 스킬 JSON을 확인한다.

### OpenBuilder 배포 전

- 현재 정상 블록·발화·스킬 연결을 스냅샷으로 기록한다.
- 폴백 블록 `/skill` 연결을 확인한다.
- 삭제 발화 반영이 필요한 경우 전체 배포 여부를 확인한다.
- 배포 시작 전 체크리스트를 완료한다.

### 운영 배포 후

- 실제 그룹방 핵심 발화 재검사
- 30분·24시간 오류율 확인
- 저장 성공·중복 차단·파서 실패 현황 확인
- 이상 시 신규 기능을 끄고 직전 정상 Worker 패키지로 재배포할 수 있게 준비한다.
