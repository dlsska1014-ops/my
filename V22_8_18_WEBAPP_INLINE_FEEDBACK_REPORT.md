# V22.8.18 웹앱 속도·피드백·로딩 개선 보고서

## 배경

웹앱-속도피드백-개선지침서가 세 문제를 제시했다.
- (A) 페이지가 느리다 → 캐싱·병렬조회·페이지네이션
- (B) 실행됐는지/왜 실패했는지 헷갈린다 → 제출 위치 인라인 결과
- (C) 처리 중인지 모른다 → 스피너·버튼 잠금·스켈레톤

## 먼저 측정 (지침서 0장)

이 저장소는 13,000줄대 단일 Worker의 성숙본으로, (A)와 (C-1)은 이미 구현되어 있었다.
- A-1 정적 자산 분리: 버전형 `/assets/*.css`·`*.js`를 `public, max-age=31536000, immutable`로 서빙, HTML은 링크만 참조
- A-3 병렬 조회: `await Promise.all` 33곳으로 페이지 로드 조회를 병렬화
- A-4 페이지네이션: 모바일 홈이 최근 10건 기본 + `전체 N건 조회` 지연 확장
- C-1 로딩: `guidedUiUxClientMain`이 POST 폼 제출 시 버튼 잠금·스피너·`aria-busy`·중복 제출 차단·bfcache 복원 처리

따라서 실제 공백은 (B)와 (B-0)이었고 그 부분에 집중했다.

## 구현 (B) 제출 위치 인라인 결과

`inlineActionResultClientMain`을 `attachUiUxRuntime` 단계에서 인라인 주입한다.
- POST 제출 직전 form action과 시각을 `sessionStorage`에 기록
- redirect로 돌아온 화면에서 상단 결과 배너(`.notice.error`/`.error`/`.err`/`.notice.ok`/`.ok`)를 탐지
- 제출한 action과 유일하게 일치하는 POST 폼을 찾으면 그 버튼 바로 아래로 배너 노드를 이동, 없거나 모호하면 상단 유지
- 결과가 화면 밖이면 `scrollIntoView`로 그 자리로 끌어옴(실패 사유를 반드시 보게)
- 성공은 3.5초 후 옅게 정리, 실패는 유지, `role`(alert/status) 부여

안전 원칙:
- 서버가 이미 `escapeHtml`로 렌더한 노드를 **이동만** 하고 새 마크업을 만들지 않음 → `innerHTML` 미사용, XSS 안전
- JavaScript·`sessionStorage` 미지원 시 기존 상단 표시로 폴백(기능 삭제 없음)
- 셸 CSS가 마지막 스타일 캐스케이드라는 다크모드 보호 규칙을 지키려 인라인 스타일을 셸 링크 앞에 삽입
- `prefers-reduced-motion` 사용자에겐 애니메이션·스무스 스크롤 비활성

## 구현 (B-0) 응답 규약

`message`가 없던 실패 JSON 13곳에 사용자용 한국어 `message`와 분기용 `reason`을 추가했다(권한·인증·검증·미발견·레이트리밋·CSRF·스코프 불일치 등). HTTP 상태 코드는 그대로 유지했다.

## 검증

- UX 55개 → 69개(+14): 인라인 런타임 존재·XSS 안전·오프스크린 스크롤·모호 폼 미이동·모바일 홈 포함·셸 캐스케이드 보존·응답 규약 message
- 영수증 55개 → 56개(+1): 영수증 페이지에 추가된 인라인 런타임의 브라우저 문법 컴파일 확인
- 전체 414개 → 429개, 개인 홈 35KB 예산 유지(25.7KB)

## 하지 않은 것 (지침서 "하지 말 것" 준수)

- 실패를 조용히 넘기는 코드 잔존 없음(JSON 실패는 message 포함)
- `innerHTML`로 사용자·서버 메시지 삽입 없음
- 결과를 화면 상단 고정 위치에만 띄우지 않음(제출 자리로 이동)
